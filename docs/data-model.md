# Pawsear Data Model

## Decision

The first Pawsear data model is designed for a Go backend using SQLite as the local-first database. The preferred implementation path is:

- SQLite schema managed through migrations.
- `sqlc` generated Go code for typed queries.
- A small domain service layer in Go for workflow rules, payment allocation, status transitions, and request conversion.
- No heavy ORM for the MVP.

The schema should stay explicit and portable. SQL should live in migration and query files, not scattered through handlers.

## Modeling Principles

- Household is the main operational grouping unit.
- Pets belong to households, not directly to one fixed owner.
- Contacts are linked to households through contextual roles.
- Payments are independent money receipts and can be allocated to many charges.
- Bookings can include many pets.
- Messages are treated as source context, not as automatic truth.
- Automated detections must remain reviewable before becoming real bookings.
- Important status changes should be explicit and auditable enough for a small business.

## SQLite Conventions

- Primary keys: `TEXT` IDs generated by the application, preferably UUIDv7 or ULID.
- Timestamps: `TEXT` ISO-8601 UTC values.
- Money: integer minor units, for example cents, stored as `amount_minor INTEGER`.
- Currency: `TEXT NOT NULL DEFAULT 'MXN'` unless configured otherwise.
- Soft deactivation: use `active INTEGER NOT NULL DEFAULT 1` where records should remain historically visible.
- Foreign keys should be enabled with `PRAGMA foreign_keys = ON`.
- Enums are stored as `TEXT` with check constraints where useful.
- Keep `created_at` and `updated_at` on business tables.
- Use `ON DELETE RESTRICT` for historical business records by default.
- Use nullable foreign keys when the real world can be unknown at creation time.
- Avoid cascading deletes for bookings, charges, payments, and messages.

## Core Tables

### households

Represents a house, family, or operational group. This is the main anchor for pets, contacts, bookings, charges, and balances.

| Column | Type | Notes |
| --- | --- | --- |
| id | TEXT PK | Application-generated ID |
| display_name | TEXT NOT NULL | Internal name, for example "Casa de Luna y Max" |
| address_line1 | TEXT | Main address |
| address_line2 | TEXT | Apartment, references, access notes if needed |
| neighborhood | TEXT | Useful for routing and scheduling |
| city | TEXT | Default city can be configured later |
| timezone | TEXT | Defaults to business timezone |
| notes | TEXT | Household-level notes |
| active | INTEGER NOT NULL DEFAULT 1 | Keeps history without deleting |
| created_at | TEXT NOT NULL | UTC |
| updated_at | TEXT NOT NULL | UTC |

Indexes:

- `idx_households_display_name`
- `idx_households_active`

### contacts

Represents a person who can message, authorize, hand off a pet, receive updates, pay, or act as emergency contact.

| Column | Type | Notes |
| --- | --- | --- |
| id | TEXT PK | Application-generated ID |
| display_name | TEXT NOT NULL | Human name |
| phone | TEXT | Normalized phone when possible |
| whatsapp_id | TEXT | Optional external identifier |
| telegram_id | TEXT | Optional external identifier |
| email | TEXT | Optional |
| notes | TEXT | Contact-level notes |
| active | INTEGER NOT NULL DEFAULT 1 | Keeps history |

Indexes:

- `idx_contacts_display_name`
- `idx_contacts_phone`
- `idx_contacts_whatsapp_id`
- `idx_contacts_telegram_id`

### household_contacts

Many-to-many relationship between households and contacts. The role belongs to this relationship, not only to the contact.

| Column | Type | Notes |
| --- | --- | --- |
| household_id | TEXT NOT NULL FK | References `households.id` |
| contact_id | TEXT NOT NULL FK | References `contacts.id` |
| role | TEXT NOT NULL | owner, partner, family, domestic_worker, payer, emergency_contact, vet, other |
| is_primary | INTEGER NOT NULL DEFAULT 0 | Primary contact for this role/context |
| notes | TEXT | Relationship-specific notes |
| created_at | TEXT NOT NULL | UTC |

Primary key:

- `(household_id, contact_id, role)`

Indexes:

- `idx_household_contacts_contact_id`
- `idx_household_contacts_role`

## Payments And Accounting

Represents money owed for completed or billable work.

| Column | Type | Notes |
| --- | --- | --- |
| id | TEXT PK | Application-generated ID |
| household_id | TEXT NOT NULL FK | References `households.id` |
| booking_id | TEXT FK | References `bookings.id`; nullable for adjustments |
| description | TEXT NOT NULL | Human-readable line item |
| amount_minor | INTEGER NOT NULL | Positive amount in minor currency units |
| currency | TEXT NOT NULL DEFAULT 'MXN' | ISO currency |

Represents money received. It does not need to belong to an owner or a single pet.

| received_at | TEXT NOT NULL | UTC |
| amount_minor | INTEGER NOT NULL | Positive amount in minor currency units |
| currency | TEXT NOT NULL DEFAULT 'MXN' | ISO currency |
| method | TEXT NOT NULL | cash, bank_transfer, card_external, other |

| updated_at | TEXT NOT NULL | UTC |

- `idx_payments_received_at`

### payment_allocations

Allocates received payment money to one or many charges.

| Column | Type | Notes |
| --- | --- | --- |
| id | TEXT PK | Application-generated ID |
| payment_id | TEXT NOT NULL FK | References `payments.id` |
| charge_id | TEXT NOT NULL FK | References `charges.id` |
| amount_minor | INTEGER NOT NULL | Positive allocated amount |
| created_at | TEXT NOT NULL | UTC |

Indexes:

- `idx_payment_allocations_payment_id`
- `idx_payment_allocations_charge_id`

Recommended uniqueness:

- `UNIQUE(payment_id, charge_id)` unless split allocations to the same charge become necessary.

Rules enforced in the Go domain service:

- A payment cannot allocate more than its total amount.
- A charge cannot receive allocations above its amount unless explicitly overpayment-supported later.
- Charge status is recalculated after allocation changes.
- Payment allocation changes should happen in a database transaction.
- Payment and charge currency must match before allocation.

### detected_requests

Reviewable request extracted from a message before it becomes a booking.

| Column | Type | Notes |
| --- | --- | --- |

Indexes:

- `idx_detected_requests_message_id`
- `idx_detected_requests_converted_booking_id`

### booking_sources

Links a booking to the message or detected request that produced it. This keeps `bookings` independent from message-capture tables and avoids forcing every manually created booking to carry messaging fields.

| Column | Type | Notes |
| --- | --- | --- |
| id | TEXT PK | Application-generated ID |
| booking_id | TEXT NOT NULL FK | References `bookings.id` |
| message_id | TEXT FK | References `messages.id` |
| detected_request_id | TEXT FK | References `detected_requests.id` |
| source_note | TEXT | Human-readable context |
| created_at | TEXT NOT NULL | UTC |

Indexes:

- `idx_booking_sources_booking_id`
- `idx_booking_sources_message_id`
- `idx_booking_sources_detected_request_id`

Rules:

- At least one of `message_id` or `detected_request_id` should be present.
- A booking can have multiple source records if later context is added, but the first source should be preserved.

## Domain Services In Go

The following logic should live in Go services, not only in SQL queries:

### BookingService

- Create booking.
- Confirm booking.
- Start booking.
- Complete booking.
- Cancel booking.
- Attach pets to booking.
- Create default care tasks from active routines.

### CareTaskService

- Complete care task.
- Skip care task with reason.
- List today's care tasks.
- List overdue care tasks.

### PaymentService

- Create charge.
- Register payment.
- Allocate payment to charges.
- Recalculate charge status.
- Validate partial payments and over-allocation.

### MessageImportService

- Import pasted message.
- Match or create conversation.
- Create detected request.
- Convert detected request to booking after human review.

## Key Reports Supported By This Model

- Today's bookings.
- Today's care tasks.
- Overdue care tasks.
- Active boarding stays.
- Household balance.
- Unpaid and partially paid charges.
- Payment history by contact.
- Completed services by date range.
- Staff workload by date range.

## Deferred Modeling

These are intentionally left out of the first schema:

- Marketplace profiles.
- Customer accounts.
- Payment processor objects.
- Subscription billing.
- Staff payroll.
- Structured recurrence rules.
- Multi-business tenant model.
- Full audit event stream.
- GPS tracking.

The model leaves room for these later without making the MVP carry their complexity now.
